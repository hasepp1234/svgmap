# サバゲーフィールドDB 自動更新パイプライン (v1)

事業計画のPhase 2「自動更新パイプライン構築」に対応するコード一式です。
月2〜3時間の運用工数に抑えるため、**「全件を人が見る」のではなく「変化した差分だけをレビュー」**する設計になっています。

## 構成

| ファイル | 役割 |
|---|---|
| `crawl.py` | 対象URLを巡回し本文テキストを取得する本番用クローラー（要インターネットアクセス環境） |
| `extractor.py` | 本文テキストから料金・レンタル・屋内外タイプを抽出するv1抽出器（正規表現ベース、APIキー不要） |
| `diff_check.py` | 既存DB(fields.json)と最新抽出結果を比較し、レビューが必要な差分だけを抽出 |
| `run_pipeline_demo.py` | 実際に取得した4フィールドの公式サイトのテキスト(`fixtures/`)を使った動作デモ |
| `fixtures/` | 動作検証用に実際の公式サイトから取得したサンプル本文（ISGF・福岡サバゲーランド・北海道サバイバルゲームパーク・東京サバゲパーク） |

## 実行済みの検証結果

`run_pipeline_demo.py` を実行し、上記4フィールドについて実際の公式サイトの本文から料金情報を抽出。
既存DBで「要公式サイト確認」となっていた4件について、具体的な金額を検出し `review_queue.csv` に出力。
その内容を確認のうえ承認し、`fields.json` / xlsx / プロトタイプサイトへ反映済みです（このサイクルが毎月の運用作業に相当します）。

## 本番運用への移行手順

1. **ホスティング環境の用意**：GitHub Actions（無料枠）等、定期実行かつ通常のインターネットアクセスができる環境を用意する。
   （このサンドボックス環境はアウトバウンド通信が制限されているため、`crawl.py`単体はここでは実行できません。`pip install requests beautifulsoup4` の上、GitHub Actions等で実行してください。）
2. **対象URLリストの拡充**：`urls.csv`（field_name, url）に巡回対象を追加していく。既存の一覧サイト（東京サバゲーナビ、サバゲーユニオン等）から一次情報(公式サイト)のURLをリサーチし、候補を増やすことで300〜500件規模へ拡張する。
3. **月次実行**：GitHub Actionsのcronで `crawl.py` → `extractor.py` → `diff_check.py` を月1回自動実行し、`review_queue.csv` を生成。
4. **人によるレビュー（月2〜3時間）**：`review_queue.csv` の差分のみを確認し、承認したものをDBに反映するスクリプト（`apply_approved.py` 相当、今後実装）を用意する。
5. **抽出精度の向上（任意）**：`extractor.py` 内の `extract_with_llm()` にAnthropic APIキーを設定すると、正規表現では拾いきれない複雑な料金体系や表記ゆれもLLMで構造化抽出できるようになる（現状はスタブ実装）。

## v1抽出器の既知の限界

正規表現ベースのため、"営業時間"のように見出しとテキストが近接しているだけで誤検出するケース（福岡サバゲーランドの例）や、屋内/屋外の判定キーワードが本文中にない場合に「要確認」となるケースがあります。これらはLLM抽出への切り替えで改善が見込めます。
